## 총평

1차 결과는 실사용 기준으로 실패입니다. 평균 combined score 0.099이며, 8건 중 정상에 가까운 케이스는 `candidate_22` 1건뿐입니다. 나머지 7건은 상품명 또는 최종단가 중 하나만 틀린 수준이 아니라, 사용자가 수정 없이는 신뢰할 수 없는 결과입니다.

다만 테스트셋 자체에도 중대한 문제가 있습니다. `selected_cases.json`의 `expectedName` 값이 한글 깨짐 상태로 저장되어 있어 자동 name score가 실제 품질을 정확히 반영하지 못합니다. 이 상태에서는 상품명 평가가 OCR 품질 평가와 정답 데이터 인코딩 오류를 동시에 측정하게 됩니다. 먼저 정답 데이터를 정상 한글로 복구해야 합니다.

## 개선 우선순위

1. 정답 데이터부터 복구
   - `selected_cases.json`의 `expectedName`을 정상 한글로 저장해야 합니다.
   - 현재 자동 name score는 신뢰도가 낮습니다.

2. 가격 후보 필터를 강화
   - `100g당`, `g당`, `정상가`, `판매가`, 바코드 하단 숫자, 날짜, 중량, 상품코드 주변 숫자는 감점 또는 제외해야 합니다.
   - 쉼표 포함 가격은 `4,480`처럼 전체 토큰으로 보존해야 하며, `480` 같은 부분 숫자 선택을 금지해야 합니다.
   - 같은 태그 안에서 가장 큰 숫자, 붉은색/굵은 글씨, `가격(원)` 또는 `원` 근처 숫자에 가중치를 줘야 합니다.

3. 가격표 영역 감지 후 OCR
   - 전체 이미지 OCR 후 파싱하는 방식은 배경 상품, 주변 안내판, 여러 가격표 때문에 불안정합니다.
   - 먼저 흰색/회색 가격표 박스 또는 전자가격표 프레임을 감지하고, 해당 영역을 crop한 뒤 OCR해야 합니다.

4. 다중 가격표 선택 규칙 추가
   - `targetRule`이 `center price tag`, `largest foreground price tag`인 케이스는 이미지 중앙성, 면적, 선명도, 가격 글자 크기를 조합해 목표 태그를 먼저 고르는 단계가 필요합니다.
   - `candidate_08` 같은 이미지는 개별 전자가격표 3개를 분리하지 않으면 계속 주변 가격을 고를 가능성이 높습니다.

5. 상품명 후보 정제
   - 상품명은 가격보다 위쪽, 한글/영문/숫자/중량 단위 조합, 2자 이상 의미 토큰 중심으로 선택해야 합니다.
   - 광고 문구, 행사기간, 원산지 단독, 바코드 코드, QR 안내 문구는 blacklist 또는 감점 처리해야 합니다.
   - OCR 깨짐 비율이 높은 문자열은 상품명으로 채택하지 말고 낮은 confidence로 반환해야 합니다.

6. 이미지 품질 게이트 도입
   - 반사, 원근 왜곡, 작은 글자, 저해상도, 목표 가격표 과소 영역 이미지는 `needs_review`로 분류해야 합니다.
   - 실패를 조용히 오답으로 내는 것보다 “자동 추출 불가”를 반환하는 편이 사용자 경험상 낫습니다.
